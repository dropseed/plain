# OpenAPI Conformance Fixture

Runs [schemathesis](https://schemathesis.readthedocs.io/) against a minimal Plain app to verify that the OpenAPI spec generated by `plain api generate-openapi` matches the runtime behavior of `plain server`.

## Layout

- `app/settings.py`, `app/urls.py` — a minimal Plain server that mounts `APIRouter` under `/api/`
- `app/api.py` — the API surface, an in-memory dict store, and the `APIRouter` with `securitySchemes`, shared `parameters`, shared `responses`, schema components, and 201 → detail OpenAPI `links`
- `run` — generates the spec, boots `plain server`, runs schemathesis, and tears the server down

## What's exercised

- CRUD on `/notes/` (`GET` list with `q`/`limit`, `POST`, `GET/PUT/PATCH/DELETE` detail)
- Path converters: `<int:>` (notes detail) and `<uuid:>` (`/echo-uuid/<uuid:id>/`)
- Required header parameter (`X-Demo-Header` on `/header-echo/`)
- Bearer-token auth via `APIKeyView` on `/secret/` — hardcoded key `conformance-key`
- API versioning via `VersionedAPIView` on `/greeting/` (`API-Version: v1` vs `v2`)
- Status codes 200, 201, 204, 400, 401, 404 across the surface
- Component refs (`schemas.{Note,NoteInput,NotePatch,Error}`, `parameters.Limit`, `responses.{BadRequest,Unauthorized,NotFound}`)
- OpenAPI `links` from `POST /notes/` 201 to `NoteDetailAPIView_{get,put,patch,delete}` so schemathesis can chain stateful tests

## How it runs

`scripts/test --server` (after Agent A's coordinated change wires this in) invokes `tests/conformance/run`. The runner:

1. Generates the OpenAPI spec into `/tmp/plain-api-conformance.json`
2. Starts `plain server` with `app.settings` in a background process on `127.0.0.1:18766`
3. Waits for the listener to come up
4. Runs `uvx --from "schemathesis>=3.36,<4" schemathesis run --checks all --header "Authorization: Bearer conformance-key" …`
5. Kills the server regardless of outcome
6. Exits with schemathesis's exit code

## Running it directly

```bash
plain-api/tests/conformance/run
```

Override the port with `PLAIN_API_CONFORMANCE_PORT=<port>`. Extra args are forwarded to `schemathesis run`:

```bash
plain-api/tests/conformance/run --hypothesis-seed=42 --hypothesis-max-examples=20
```

## Hardcoded API key

`/secret/` requires `Authorization: Bearer conformance-key`. The `run` script passes this header to schemathesis so any operation it generates against secured paths is accepted. There is no `APIKey` model — the fixture short-circuits `APIKeyView.get_api_key()` to compare against a hardcoded string and avoid pulling in `plain.postgres`.

## Framework assumptions

The fixture is written against the post-coordinated-change framework:

- Default `operationId` is `{ViewClass}_{method}` (used in the `links` block on the `POST /notes/` 201)
- `request.json_data` raises `BadRequestError400` for both invalid JSON and wrong/missing content-type
- `ErrorSchema` is `{id, message, errors?}` — no `url` field
