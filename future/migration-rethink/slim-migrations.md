# Slim migrations

## Problem

Migration files currently contain everything: schema changes, index operations, constraint additions, NOT NULL alterations. With schema convergence handling indexes/constraints/NOT NULL, migrations can be stripped down to only the operations that are truly imperative and code-dependent.

## What changes

`makemigrations` stops generating:

- `AddIndex` / `RemoveIndex`
- `AddConstraint` / `RemoveConstraint` (FK, CHECK, UNIQUE)
- `AlterField` for null/not-null changes
- `AlterField` for db_index changes

`makemigrations` continues generating:

- `CreateModel` / `DeleteModel`
- `AddField` / `RemoveField`
- `RenameField` / `RenameModel`
- `AlterField` for column type changes
- `RunPython` / `RunSQL` (data operations, explicitly written by developer)

### CreateModel behavior

`CreateModel` generates a `CREATE TABLE` with columns, types, and NOT NULL — but no FK constraints, CHECK constraints, indexes, or unique constraints. Those are all convergence concerns.

```sql
-- 20240101_120000_create_orders.sql
CREATE TABLE orders (
    id bigint NOT NULL GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    status text NOT NULL,
    author_id bigint NULL
);
```

NOT NULL is part of the column definition in CREATE TABLE, not a separate constraint operation. On a new table with zero rows, there's nothing to check — it's free. This isn't inconsistent with AddField (which creates nullable columns for convergence to handle later) — it's recognizing that CREATE TABLE and ALTER TABLE have different semantics. A new table has no data; an existing table might.

Convergence runs after migrations and adds the FK constraint on `author_id`, any indexes, and any other declared constraints. On a fresh table this is fast — zero rows to scan.

### AddField behavior change

When adding a FK field like `author = ForeignKeyField(User)`, today's schema editor creates the FK constraint **inline in the column DDL** (`ADD COLUMN author_id bigint REFERENCES users(id) DEFERRABLE INITIALLY DEFERRED`) and queues deferred SQL for index creation. The FK constraint is part of the same statement that creates the column.

With slim migrations:

- AddField generates `ALTER TABLE ADD COLUMN author_id bigint NULL` — just the column, nullable, no constraint, no FK reference
- The FK constraint, index, and NOT NULL are all convergence concerns
- Convergence adds the FK constraint separately using `ADD CONSTRAINT ... NOT VALID` then `VALIDATE CONSTRAINT`

This means AddField no longer needs a dependency on the target model's migration. The column is just a bigint. The target table doesn't even need to exist yet — convergence will add the FK constraint after all migrations have run and all tables exist.

## Migration file format

Schema migrations are `.sql` files. Data migrations are `.py` files. Both live in the same directory, sorted by timestamp.

### Schema migrations (`.sql`)

Auto-generated by `postgres schema --make`. Pure SQL, no wrapper.

A typical migration goes from today's Python operation format:

```python
# OLD FORMAT (current)
class Migration:
    dependencies = [('users', '0003_add_fields')]
    operations = [
        AddField('order', 'author', ForeignKeyField(to='users.User')),
        AddIndex('order', Index(fields=['author'], name='orders_author_idx')),
        AddIndex('order', Index(fields=['created_at'], name='orders_created_idx')),
        AlterField('order', 'status', TextField(null=False)),
    ]
```

To:

```sql
-- 20240315_110000_add_author_to_orders.sql
ALTER TABLE orders ADD COLUMN author_id bigint NULL;
```

The rest — indexes, FK constraint, NOT NULL — lives on the model and converges.

Each `.sql` file contains a single DDL statement. For multi-statement operations, use a `.py` data migration.

### Data migrations (`.py`)

Created by `postgres schema --make --empty`. Developer-written Python for backfills, seed data, or custom SQL.

The migration runs with Plain fully configured, so you import models directly and use the ORM — no historical model reconstruction:

```python
# 20240320_090000_backfill_status.py
class Migration:
    def run(self, connection):
        from app.models import Order
        Order.query.filter(status=None).update(status="active")
```

Raw SQL is also available via `connection`:

```python
# 20240320_090000_normalize_emails.py
class Migration:
    def run(self, connection):
        with connection.cursor() as cursor:
            cursor.execute("UPDATE users SET email = LOWER(email) WHERE email != LOWER(email)")
```

### ORM vs raw SQL in data migrations

Data migrations use the **current** model code, not a historical snapshot. This means if you write a migration referencing `Order.status` and someone later renames the field to `Order.state`, the old migration breaks.

In practice this is fine because:

- Data migrations are short-lived — backfills become no-ops on empty tables and can be deleted once all databases have applied them
- The failure is loud (AttributeError, not silent corruption)
- Django's historical models were "pale shadows" anyway — no custom methods, no managers, no properties — so most developers already used real imports or raw SQL

**When to use ORM:** one-time backfills you'll delete soon. The ORM is nicer to write and read.

**When to use raw SQL:** long-lived migrations that might outlive the model code they reference (seed data, operations shared across many environments that won't all be updated at once). Raw SQL references table/column names, which are more stable than Python attribute names.

### Runner behavior

The runner globs for `*.sql` and `*.py`, extracts timestamps from filenames, sorts by timestamp, and runs in order within the batch transaction:

- `.sql` files: execute the file contents directly
- `.py` files: import the module, call `Migration.run(connection)`

### Fresh-db discrimination

File extension determines behavior on fresh databases:

- `.sql` = schema = **skip** (schema already exists from model-based DDL)
- `.py` = data = **run** (seed data, backfills needed for app to function)

Backfill migrations that touch existing data are no-ops on empty tables. See fresh-db-from-models.md.

### Transition from current format

The runner also supports the current `.py`-with-operations format (Django-style). Existing migrations keep working. New migrations use the new format. Over time, old migrations get deleted as projects adopt fresh-db-from-models.

### Why `.sql` for schema

- SQL syntax highlighting in editors and GitHub PR diffs
- Can copy-paste into psql to debug
- No Python boilerplate for the common case
- `makemigrations` implementation is trivial: write SQL to a file
- Matches Atlas, Prisma, Flyway, and goose — every modern tool that auto-generates migrations produces `.sql`

### Metadata

The rare `after` ordering hint goes in a SQL comment:

```sql
-- after: 20230501_000000
ALTER TABLE orders ADD COLUMN status text NULL;
```

Or use a `.py` file if you need richer metadata alongside schema SQL:

```python
class Migration:
    after = "20230501_000000"
    sql = "ALTER TABLE orders ADD COLUMN status text NULL;"
```

### The nullable window tradeoff

With PG 11+, `ALTER TABLE ADD COLUMN status text NOT NULL DEFAULT 'active'` is catalog-only and instant — Postgres stores the default in `pg_attribute.attmissingval` and old code's INSERTs automatically get the default. No NULLs ever appear. Single deploy to final state.

In the slim model, AddField creates a nullable column. Convergence later adds `db_default`, backfills NULLs, and applies NOT NULL. Between the migration and convergence, the column accepts NULLs.

**This is a deliberate tradeoff: zero-downtime safety over single-deploy finality.**

For non-rolling deploys (single server, dev, test), all steps happen in the same `postgres sync` run — the nullable window is negligible. For rolling deploys where old code runs alongside new code, the expand-and-contract pattern is what you want — old code doesn't know about the column, and convergence waits until all code is updated before tightening constraints.

### Column type changes

`makemigrations` generates ALTER COLUMN TYPE with a warning comment when the change could be lossy:

```sql
-- 20240320_090000_widen_status.sql
-- Type change: orders.status varchar(255) → text (safe, widening)
ALTER TABLE orders ALTER COLUMN status TYPE text;
```

```sql
-- 20240320_090000_change_amount_type.sql
-- ⚠ Type change: orders.amount text → integer (may lose data, add USING clause)
ALTER TABLE orders ALTER COLUMN amount TYPE integer; -- USING amount::integer
```

The developer reviews the generated `.sql` file and edits if needed (e.g., adding a `USING` clause for the cast). This is the same review workflow Atlas and Prisma use — the tool generates, the developer verifies.

### RenameModel/RenameField and convergence

A rename migration generates `ALTER TABLE RENAME TO` or `ALTER TABLE RENAME COLUMN`. Postgres renames the primary key constraint automatically but NOT other indexes or constraints.

After a table rename from `orders` to `purchases`, convergence sees:

- `orders_user_id_idx` exists in DB but no model declares it → unmanaged
- `purchases_user_id_idx` declared on model but doesn't exist → create it

Convergence creates the new indexes/constraints and the old ones become stale (reported by `postgres schema`, dropped with `--prune`). This means indexes get rebuilt after a rename — a cost, but renames are rare. The alternative (tracking renames inside convergence to use `ALTER INDEX RENAME TO`) adds complexity for a case that almost never happens.

## Considerations

- Need to handle the transition: existing projects have `.py` migrations with operation objects. The runner supports both formats.
- `RemoveField` on a FK column: Postgres cascades the FK constraint and index automatically when the column is dropped, so convergence doesn't need to clean up first.
