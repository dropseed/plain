# Slim migrations

## Problem

Migration files currently contain everything: schema changes, index operations, constraint additions, NOT NULL alterations. With schema convergence handling indexes/constraints/NOT NULL, migrations can be stripped down to only the operations that are truly imperative and code-dependent.

## What changes

`migrations create` stops generating:

- `AddIndex` / `RemoveIndex` — **already done**
- `AddConstraint` / `RemoveConstraint` (FK, CHECK, UNIQUE) — **already done**
- `AlterField` for null/not-null changes
- `AlterField` for db_index changes

`migrations create` continues generating:

- `CreateModel` / `DeleteModel`
- `AddField` / `RemoveField`
- `RenameField` / `RenameModel`
- `AlterField` for column type changes
- `RunPython` / `RunSQL` (data operations, explicitly written by developer)

### CreateModel behavior

`CreateModel` generates a `CREATE TABLE` with columns, types, and NOT NULL — but no FK constraints, CHECK constraints, indexes, or unique constraints. Those are all convergence concerns.

```sql
-- 20240101_120000_create_orders.sql
CREATE TABLE orders (
    id bigint NOT NULL GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    title text NOT NULL,
    status text NOT NULL,
    author_id bigint NULL
);
```

NOT NULL is part of the column definition in CREATE TABLE, not a separate constraint operation. On a new table with zero rows, there's nothing to check — it's free. This isn't inconsistent with AddField (which creates nullable columns for convergence to handle later) — it's recognizing that CREATE TABLE and ALTER TABLE have different semantics. A new table has no data; an existing table might.

Convergence runs after migrations and adds the FK constraint on `author_id`, any indexes, and any other declared constraints. On a fresh table this is fast — zero rows to scan.

### AddField behavior change

When adding a FK field like `author = ForeignKeyField(User)`, today's schema editor creates the FK constraint **inline in the column DDL** (`ADD COLUMN author_id bigint REFERENCES users(id) DEFERRABLE INITIALLY DEFERRED`) and queues deferred SQL for index creation. The FK constraint is part of the same statement that creates the column.

With slim migrations:

- AddField generates `ALTER TABLE ADD COLUMN author_id bigint NULL` — just the column, nullable, no constraint, no FK reference
- The FK constraint, index, and NOT NULL are all convergence concerns
- Convergence adds the FK constraint separately using `ADD CONSTRAINT ... NOT VALID` then `VALIDATE CONSTRAINT`

This means AddField no longer needs a dependency on the target model's migration. The column is just a bigint. The target table doesn't even need to exist yet — convergence will add the FK constraint after all migrations have run and all tables exist.

## Migration file format

All migrations are `.py` files with a `Migration` class. Schema migrations have `operations`. Data migrations have `run()`. Both live in the same directory, sorted by timestamp.

### Schema migrations (auto-generated)

Auto-generated by `migrations create`. Each operation maps to exactly one SQL statement:

```python
# OLD FORMAT (current Django-style)
class Migration:
    dependencies = [('users', '0003_add_fields')]
    operations = [
        AddField('order', 'author', ForeignKeyField(to='users.User')),
        AddIndex('order', Index(fields=['author'], name='orders_author_idx')),
        AddIndex('order', Index(fields=['created_at'], name='orders_created_idx')),
        AlterField('order', 'status', TextField(null=False)),
    ]
```

Becomes:

```python
# NEW FORMAT — thin operations, no indexes/constraints
from plain.postgres.migrations import AddColumn

class Migration:
    operations = [
        AddColumn("orders", "author_id", "bigint NULL"),
    ]
```

The rest — indexes, FK constraint, NOT NULL — lives on the model and converges.

**Why operations, not raw .sql files:** The migration/convergence boundary is enforced by the API. There is no `CreateIndex` operation, no `SetNotNull` operation, no `AddConstraint` operation — those are convergence concerns. You can't cross the boundary without explicitly using `RunSQL`. With .sql files, users could write any DDL (CREATE INDEX, SET NOT NULL) and bypass convergence's safe DDL patterns. See thin-operations.md for the full analysis.

### Data migrations (developer-written)

Created by `migrations create --empty`. The migration runs with Plain fully configured — import models directly and use the ORM:

```python
# 20240320_090000_backfill_status.py
class Migration:
    def run(self, connection):
        from app.models import Order
        Order.query.filter(status=None).update(status="active")
```

Raw SQL is also available via `connection`:

```python
# 20240320_090000_normalize_emails.py
class Migration:
    def run(self, connection):
        with connection.cursor() as cursor:
            cursor.execute("UPDATE users SET email = LOWER(email) WHERE email != LOWER(email)")
```

### No historical model reconstruction

The current `RunPython` receives `(models_registry, schema_editor)` where `models_registry` is a reconstructed historical model registry built by replaying state mutations from every prior migration. This is why the autodetector generates migrations for changes that produce no DDL (`choices`, `validators`, `default`, `on_delete`, `related_name`, `ordering`) — they exist solely to keep the reconstructed state accurate.

The reconstructed models are barely useful: no custom methods, managers, or properties. Most developers import the real model class anyway. And the accuracy goal is questionable — reconstructed state silently drifts from reality in subtle ways.

Data migrations should use real imports. The `run(self, connection)` signature replaces `RunPython(fn)` with `fn(models_registry, schema_editor)`. Once nothing consumes the historical state, the autodetector stops generating state-only migrations, and the entire `ModelState`/`ProjectState` replay machinery becomes dead weight that can be removed incrementally.

### ORM vs raw SQL in data migrations

Data migrations use the **current** model code, not a historical snapshot.

**When to use ORM:** Most data migrations — backfills, data transformations, anything where the model API makes the intent clear.

**When to use raw SQL:** When you need precise control over the SQL, or when the operation doesn't map to model methods (e.g., bulk updates with complex expressions).

These migrations are incremental-only. If fresh-db-from-models is implemented later, fresh databases should not replay historical data operations at all. Application seed/init data is a separate concern and is out of scope here.

### Transition from current format

The runner also supports the current Django-style `.py` format (with `dependencies` and Field-based operations). Existing migrations keep working. New migrations use thin operations. Over time, old migrations get deleted as projects adopt fresh-db-from-models.

### Metadata

The `after` ordering hint is a Python attribute:

```python
class Migration:
    after = "20230501_000000"
    operations = [
        AddColumn("orders", "status", "text NULL"),
    ]
```

### The nullable window tradeoff

With PG 11+, `ALTER TABLE ADD COLUMN status text NOT NULL DEFAULT 'active'` is catalog-only and instant — Postgres stores the default in `pg_attribute.attmissingval` and old code's INSERTs automatically get the default. No NULLs ever appear. Single deploy to final state.

In the slim model, AddField creates a nullable column. Convergence applies `SET DEFAULT` in pass 0 (immediately after migrations), then backfills NULLs, then applies NOT NULL in later passes. This is one of two purposes pass 0 serves — the other is managing `db_default` changes on existing columns. See schema-convergence.md "Execution passes" for the full pass breakdown.

**This is a deliberate tradeoff: zero-downtime safety over single-deploy finality.**

For non-rolling deploys (single server, dev, test), pass 0 runs immediately after migrations in the same `postgres sync` — the nullable window is effectively zero (migration adds column → pass 0 applies SET DEFAULT → new rows get the correct value). For rolling deploys where old code runs alongside new code, the expand-and-contract pattern is what you want — old code doesn't know about the column, and convergence waits until all code is updated before tightening constraints.

### Python-only callable defaults

The convergence backfill path works when the default has a SQL equivalent (`'active'`, `now()`, `gen_random_uuid()`). But some fields have Python-only callable defaults with no SQL equivalent — `default=get_random_string`, custom functions, etc.

For these, convergence can't do `UPDATE ... SET col = <sql_expr>` because there's no SQL expression. Convergence should refuse to auto-backfill and tell the developer to write a data migration. The common cases (static values, `now()`, `gen_random_uuid()`) all have SQL equivalents and work automatically.

The long-term direction is to deprecate and eventually block Python callable defaults entirely, requiring DB expressions or static values. See [fields-db-defaults](../postgres-first-data-layer/fields-db-defaults.md) for the full design.

Note: the current migration framework (pre-rethink) has a related bug — `AddField` with a callable default evaluates the callable once in Python and uses the single result as a static SQL `DEFAULT` for all existing rows. This means `default=uuid.uuid4` gives every existing row the same UUID. In Plain that becomes especially visible once a later `UniqueConstraint` is declared on the field: convergence tries to add the unique constraint and fails on the duplicated backfilled values.

### Column type changes

`migrations create` generates ALTER COLUMN TYPE with a warning comment when the change could be lossy:

```sql
-- 20240320_090000_widen_status.sql
-- Type change: orders.status varchar(255) → text (safe, widening)
ALTER TABLE orders ALTER COLUMN status TYPE text;
```

```sql
-- 20240320_090000_change_amount_type.sql
-- ⚠ Type change: orders.amount text → integer (may lose data, add USING clause)
ALTER TABLE orders ALTER COLUMN amount TYPE integer; -- USING amount::integer
```

The developer reviews the generated migration and edits if needed (e.g., adding a `using=` parameter to `AlterColumnType` for the cast). The tool generates, the developer verifies.

### RenameModel/RenameField and convergence

A rename migration generates `ALTER TABLE RENAME TO` or `ALTER TABLE RENAME COLUMN`. Postgres renames the primary key constraint automatically but NOT other indexes or constraints.

After a table rename from `orders` to `purchases`, convergence sees:

- `orders_user_id_idx` exists in DB but no model declares it → unmanaged
- `purchases_user_id_idx` declared on model but doesn't exist → create it

Convergence creates the new indexes/constraints and the old ones become stale (reported by `postgres schema`, dropped with `--prune`). This means indexes get rebuilt after a rename — a cost, but renames are rare. The alternative (tracking renames inside convergence to use `ALTER INDEX RENAME TO`) adds complexity for a case that almost never happens.

## Considerations

- Need to handle the transition: existing projects have `.py` migrations with operation objects. The runner supports both formats.
- `RemoveField` on a FK column: Postgres cascades the FK constraint and index automatically when the column is dropped, so convergence doesn't need to clean up first.
