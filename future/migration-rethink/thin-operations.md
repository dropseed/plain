# Migration format and auto-detection

## Problem

Today's migration system generates migrations for model changes that don't affect the database at all:

| Change             | Migration? | Actual DDL? |
| ------------------ | ---------- | ----------- |
| `choices`          | Yes        | No          |
| `validators`       | Yes        | No          |
| `default` (non-db) | Yes        | No          |
| `on_delete`        | Yes        | No          |
| `related_name`     | Yes        | No          |
| `ordering`         | Yes        | No          |
| `required`         | Yes        | No          |
| `error_messages`   | Yes        | No          |

These exist to maintain "migration state" — a Python reconstruction of the model at each historical point. This state powers `apps.get_model()` in RunPython, which gives you a historical model with the right choices, validators, etc.

But the reconstructed model is a pale shadow: no custom methods, managers, or properties. Most developers are surprised when it doesn't work like the real model, and end up importing the actual class or using raw SQL anyway.

**Regardless of approach below, no-op migrations are eliminated.** Only DB-relevant changes produce migrations.

## Direction: Thin operations with DB introspection

Migration files use a small set of operation classes. Each operation maps to exactly one SQL statement. Auto-detection uses DB introspection (the existing `postgres schema` comparison engine) to determine what's changed.

### The operation set

```python
CreateTable(table, columns)          # CREATE TABLE ... (...);
RemoveTable(table)                   # DROP TABLE ...;
AddColumn(table, column, type)       # ALTER TABLE ... ADD COLUMN ...;
RemoveColumn(table, column)          # ALTER TABLE ... DROP COLUMN ...;
RenameColumn(table, old, new)        # ALTER TABLE ... RENAME COLUMN ... TO ...;
RenameTable(old, new)                # ALTER TABLE ... RENAME TO ...;
AlterColumnType(table, column,       # ALTER TABLE ... ALTER COLUMN ... TYPE ...
                new_type, using=None) #   [USING ...]
RunSQL(sql)                          # escape hatch — executes raw SQL
```

Eight operations. Each is ~20-30 lines of code: store the args, emit one SQL statement, update an in-memory schema dict for state replay. Total framework code: ~300-500 lines.

**What's NOT in this list:** CreateIndex, AddConstraint, SetNotNull, SetDefault. Those operations don't exist because they're convergence concerns. The migration/convergence boundary is enforced by the API — you can't cross it without RunSQL.

### Migration files

Schema migrations (auto-generated by `postgres schema --make`):

```python
# 20240315_110000_add_status_to_orders.py
from plain.postgres.migrations import AddColumn

class Migration:
    operations = [
        AddColumn("orders", "status", "text NULL"),
    ]
```

Data migrations (developer-written via `postgres schema --make --empty`):

```python
# 20240320_090000_backfill_status.py
class Migration:
    def run(self, connection):
        from app.models import Order
        Order.query.filter(status=None).update(status="active")
```

Raw SQL escape hatch (when operations don't cover the need):

```python
# 20240325_100000_custom_ddl.py
from plain.postgres.migrations import RunSQL

class Migration:
    operations = [
        RunSQL("CREATE EXTENSION IF NOT EXISTS pg_trgm"),
    ]
```

### Why thin operations over raw SQL files

The original design used `.sql` files for schema migrations. The footgun analysis changed this.

**The boundary problem with .sql files:** Users can write anything in a SQL file — `CREATE INDEX`, `SET NOT NULL`, `ADD CONSTRAINT`. These cross into convergence territory, bypassing safe DDL patterns (CONCURRENTLY, NOT VALID, CHECK-then-NOT-NULL). The framework needs a SQL linter to catch boundary violations — fragile, needs pattern matching, escape hatches, false positives.

**With operations, the boundary is the API.** There's no `CreateIndex` operation. You literally can't create an index in a migration unless you use `RunSQL` — which is explicitly marked as an escape hatch and easy to flag in code review.

| Concern                 | .sql files                        | Thin operations                             |
| ----------------------- | --------------------------------- | ------------------------------------------- |
| Boundary enforcement    | SQL linter (fragile)              | API (robust — operations don't exist)       |
| RunSQL escape hatch     | Indistinguishable from normal SQL | Explicitly marked, easy to lint             |
| DB-free CI `--check`    | Requires Postgres                 | Replay operations in memory                 |
| Code review             | SQL highlighting                  | Python, but operations are self-documenting |
| Copy-paste into psql    | Direct                            | Read the SQL from the operation             |
| Abstraction maintenance | None                              | ~300-500 lines, trivial 1:1 mapping         |

The abstraction layer is real but minimal. Each operation maps to one SQL statement with no complex translation. The maintenance burden is a few hundred lines of straightforward code. The payoff is boundary enforcement by design and DB-free CI checks.

### Auto-detection: DB introspection with drift check

`postgres schema --make` introspects the developer's database and compares against model definitions. This is the same comparison engine that `postgres schema` already uses — no new infrastructure.

Before diffing, it verifies the DB matches existing migration state (refuses with a clear error if drift is detected). This prevents generating incorrect operations from a dirty dev DB.

**Why DB introspection, not pure state replay?** Plain already has the comparison engine (`postgres schema`). Reusing it is the simplest path. State replay (computing expected state from operation files alone) is a secondary capability — useful for CI `--check` without a DB, but not the primary generation path.

### CI checks without a database

With operations, `postgres schema --check` can replay operations in memory to compute expected schema state, then compare against model definitions. No database needed.

```
# CI: verify no pending model changes (no DB required)
plain postgres schema --check

# CI: verify migrations actually work (needs DB)
plain postgres sync --replay
```

This resolves the "needs DB for generation" concern. Generation still uses DB introspection (you have a DB in dev). Validation can work without one.

## RunPython uses current code

With no historical model reconstruction, data migrations import models directly and use the ORM:

```python
class Migration:
    def run(self, connection):
        from app.models import Order
        Order.query.filter(status=None).update(status="active")
```

Or raw SQL when the ORM doesn't fit:

```python
class Migration:
    def run(self, connection):
        with connection.cursor() as cursor:
            cursor.execute("UPDATE orders SET status = 'active' WHERE status IS NULL")
```

The migration runs with Plain configured, same connection as the batch transaction. ORM operations are inside the transaction. It's the developer's responsibility to ensure data migrations work with the current codebase. Since old migrations get deleted (fresh DBs use model definitions), and data migrations are for one-time operations, this is practical.

### ORM vs raw SQL in data migrations

Data migrations use the **current** model code, not a historical snapshot. This means if you write a migration referencing `Order.status` and someone later renames the field to `Order.state`, the old migration breaks.

In practice this is fine because:

- Data migrations are short-lived — backfills become no-ops on empty tables and can be deleted once all databases have applied them
- The failure is loud (AttributeError, not silent corruption)
- Django's historical models were "pale shadows" anyway — no custom methods, no managers, no properties — so most developers already used real imports or raw SQL

**When to use ORM:** one-time backfills you'll delete soon. The ORM is nicer to write and read.

**When to use raw SQL:** long-lived migrations that might outlive the model code they reference (seed data, operations shared across many environments that won't all be updated at once). Raw SQL references table/column names, which are more stable than Python attribute names.

## What this eliminates

- No-op migrations for choices, validators, help_text, etc.
- `non_db_attrs` filtering in the schema editor
- `ModelState` / `ProjectState` / historical model reconstruction
- `apps.get_model()` in RunPython (use real imports instead)
- Full Field objects in migration files
- The entire "migration state" machinery
- The dependency graph and resolver
- The schema editor's complex DDL generation (replaced by trivial 1:1 operation → SQL mapping)

## Industry research

### Django — state replay, no DB required for generation

Django's autodetector compares two `ProjectState` objects: one built by replaying all existing migration operations in memory, the other constructed from current model classes. The diff produces operation objects (`AddField`, `RemoveField`, etc.) that both mutate state forward and generate DDL via the schema editor.

**No database needed for `makemigrations`.** The entire comparison is in-memory. This is Django's key architectural bet — and the source of its complexity.

**The cost is real and measurable.** In the Plain fork: `autodetector.py` (1,379 lines), `state.py` (882 lines), operation classes (1,528 lines across 4 files), `serializer.py` (378 lines), `writer.py` (302 lines), `loader.py` (377 lines), `graph.py` (364 lines). That's ~5,200 lines just for the state/replay/generation machinery, plus the schema editor (1,815 lines) that serves as the operation-to-DDL translation layer. In upstream Django, autodetector.py alone is 2,066 lines and state.py is 1,021 lines.

Plain's thin operations would be ~300-500 lines total — the convergence split eliminates the vast majority of Django's complexity.

### Rails, Ecto, Laravel — manual generation

All three use manual migration authoring (developer writes DSL). No auto-detection. Rails uses `schema.rb` (DB introspection dump) for fresh DBs. Ecto uses `structure.sql`. Laravel uses `schema:dump`. Three of the four major web frameworks have fully manual migration authoring. Django is the outlier with auto-detection.

### Prisma — auto-detection, .sql output, requires shadow DB

Prisma auto-detects changes and generates `.sql` migration files. Requires a "shadow database" to replay history. Cannot detect renames. This is the approach we initially considered (Option A with .sql output) — but Prisma's .sql files have the same boundary problem: users can edit them to contain any SQL.

### Atlas — auto-detection, .sql output, requires dev DB

Atlas generates SQL migration files from schema diffs. Requires a dev database (`--dev-url`). Has 50+ safety analyzers that lint the generated SQL. Atlas's linting approach is the best version of the "enforce boundaries on .sql files" pattern — but it's still pattern matching on SQL, not structural enforcement.

### Stripe's pg-schema-diff — declarative, no migration files

Purely declarative. Diffs desired state (SQL DDL) against live DB, generates a plan, applies with safe DDL. No migration files at all. Every safe DDL pattern Plain's convergence uses is validated at Stripe scale in this tool. Implements per-statement `lock_timeout` and `statement_timeout`, a hazard system that refuses to apply without explicit acknowledgment, and plan validation against a temporary database.

### EF Core — auto-detection via model snapshot, operations, no DB required

EF Core maintains a `ModelSnapshot.cs` file and generates operation-based migration files. No database connection needed. This is the closest to Plain's approach: thin operations + a state tracking mechanism. The snapshot file is a merge conflict hotspot, but the overall architecture is sound.

## Open questions

- **Resolved: migration file format.** `.py` with thin operations for schema migrations, `.py` with `run()` for data migrations. Operations enforce the migration/convergence boundary at the API level. RunSQL is the explicit escape hatch.
- **Resolved: rename detection.** Interactive prompt, same as Django and Atlas. "Did you rename `title` to `name`?" The diff sees a column disappear and another appear with the same type.
- **Resolved: CI migration check without Postgres.** Replay operations in memory to compute expected state, compare against models. No DB needed for `--check`.
- **Resolved: `makemigrations` safeguard.** Always verify DB matches migration state first (drift check). Refuse to generate if drift detected.
- **Resolved: package-shipped migrations.** Packages ship `.py` files with operations. Same format as project migrations.
