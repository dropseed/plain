# Sync behavior and semantics

How `postgres sync` works internally. For the CLI surface (flags, output format, full command list), see cli-design.md.

## What sync does

`postgres sync` brings a database in line with the current code. One command, two modes depending on database state.

### On an existing database (incremental)

1. **Acquire sync advisory lock** — serialize the full sync run against other schema-changing commands (multi-node deploy)
2. **Apply pending migrations** — in timestamp order, batch transaction (all-or-nothing). Schema operations (AddField, CreateModel) and data operations (RunPython, RunSQL) run together.
3. **Run required convergence** — only non-destructive correctness work needed for the declared contract: defaults, FK/CHECK/UNIQUE/NOT NULL, using online-safe Postgres patterns. If the model declares a correctness invariant that cannot be reached, `postgres sync` fails with an actionable error.
4. **Run performance convergence** — secondary indexes and other best-effort performance work. Failures warn and retry later.
5. **Report explicit cleanup opportunities** — anything removable that requires `--prune` on a contraction deploy

### On a fresh database (empty, no tables)

1. **Generate DDL from model definitions** — `CREATE TABLE` for all registered models, with columns, types, and defaults
2. **Run required convergence** — defaults, constraints, NOT NULL
3. **Run performance convergence** — create any declared secondary indexes
4. **Mark all migrations as applied** — prevents re-application on subsequent sync runs

Fresh databases are schema-only in this design. Historical data migrations do not replay on an empty database built from current models, and app seeding/init is a separate concern outside this doc's scope. See fresh-db-from-models.md.

## Idempotency

`postgres sync` is safe to run repeatedly:

- Migrations already applied are skipped (tracked in DB)
- Required convergence compares desired vs actual — already-correct objects are skipped
- Performance convergence compares desired vs actual — already-built indexes are skipped
- INVALID indexes from failed CONCURRENTLY builds are detected, dropped, and retried
- Partially converged state (3 of 5 indexes created, then failure) converges the remaining 2 on next run

## Production pre-deploy contract

`postgres sync` is intended to run before new application code is promoted.

- **Migration phase is atomic by default** — autogenerated migrations should be safe to run as one batch transaction. If any migration fails, the batch rolls back, the command exits non-zero, and convergence does not run.
- **Convergence is forward-only in normal sync** — the default pre-deploy path only applies monotonic work that is safe to leave partially completed: create, add, validate, attach, set, finalize. Removing undeclared objects or rebuilding existing ones is explicit maintenance, not part of ordinary `postgres sync`.
- **A correctness convergence failure aborts the deploy** — `postgres sync` exits non-zero, old code keeps running, and the operator fixes the data/state issue before rerunning. Already-applied convergence work remains committed.
- **Cleanup is a separate step** — drops and other contraction work require `postgres sync --prune` (or another explicit maintenance command), never the default pre-deploy path.

One important nuance: "online-safe" is not the same as "rollout-safe". `postgres sync` can safely execute `SET NOT NULL` with the right Postgres pattern, but the schema author still has to declare `null=False` only when old code is already compatible with it.

## Two deploy shapes

Most deploys should be boring:

```bash
plain postgres sync
```

That is the normal additive/tightening deploy path. It applies migrations, required correctness convergence, and best-effort performance work. It does **not** remove previously declared objects.

Contraction deploys are rarer:

```bash
plain postgres sync --prune
```

Use `--prune` when the deploy intentionally removes convergence-owned indexes or constraints that are no longer declared in the model contract. This is not a special deployer trick; it is the explicit signal that this deploy includes contraction work.

So the current operating model is:

- **ordinary deploy** = `plain postgres sync`
- **contraction deploy** = `plain postgres sync --prune`

If Plain becomes deploy-aware later, this explicit contraction step could be automated. For now, `--prune` is the simplest honest contract. See deploy-aware-rollouts.md.

## Advisory lock coordination

Multiple sync processes (e.g., during a rolling deploy) are serialized by a Postgres advisory lock. The lock covers the full `postgres sync` contract — migrations plus required/performance convergence — so one process cannot report a successful sync against stale pre-migration state while another is still changing the schema.

If a sync process crashes while holding the lock, the lock is released automatically (session-level advisory lock). No manual intervention needed.

## The batch transaction for migrations

All pending migrations run in a single transaction. If any migration fails, all roll back. For schema migrations, the runner executes each operation's generated SQL. For data migrations, it calls `run(connection)`. This works because slim migrations are catalog-only DDL (fast, no long locks) plus data operations. No CONCURRENTLY or NOT VALID operations in migrations — the operation set doesn't include them, and they're convergence concerns.

This is a deliberate tradeoff: you can't have a partially-applied migration set. Either all pending migrations apply or none do. This eliminates the "stuck between migrations" state that plagues Django deployments with `atomic=False`.

If the framework keeps an escape hatch for bespoke non-atomic migrations, it should stay outside the default `postgres sync` contract. Normal generated migrations should remain atomic-only.

## Convergence ordering

Convergence uses a fixed six-pass execution order (pass 0 through pass 5). Pass 0 applies defaults immediately after migrations to close the nullable window — new rows get the correct default before indexes and constraints are built. See schema-convergence.md "Convergence ordering" for the full pass breakdown and lock budget.

## Considerations

- Should `postgres sync` be the default action of `plain postgres` with no subcommand? (Probably not — accidental writes are worse than an extra word.)
- Should `plain check` (preflight) include a `postgres sync --check`? (Yes — see cli-design.md preflight integration.)
