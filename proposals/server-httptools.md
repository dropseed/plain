# Replace pure Python HTTP parser with a native parser

## Context

Plain's HTTP parser (`server/http/message.py`, `server/http/parser.py`) is vendored from gunicorn — a pure Python parser that reads from the socket via the `Unreader` abstraction, scans for `\r\n`, and parses request lines and headers in Python. It's correct and battle-tested, but slow compared to native alternatives.

The main concrete technical advantage ASGI servers (Uvicorn) have over Plain's integrated server is parse performance. Adopting a native parser neutralizes this while keeping all the architectural benefits of the integrated server.

## Options

### Option 1: httptools (Cython + llhttp/C)

[httptools](https://github.com/MagicStack/httptools) is a Cython wrapper around llhttp, the C HTTP parser from Node.js. It's what Uvicorn uses.

```python
class RequestProtocol:
    def on_url(self, url: bytes): ...
    def on_header(self, name: bytes, value: bytes): ...
    def on_headers_complete(self): ...
    def on_body(self, body: bytes): ...
    def on_message_complete(self): ...

parser = httptools.HttpRequestParser(RequestProtocol())
parser.feed_data(raw_bytes)  # triggers callbacks
```

- ~3x faster than pure Python parsing
- Push parser (you feed it bytes, it calls your callbacks), zero I/O
- Single dependency, no transitive deps, maintained by MagicStack (same team as uvloop)
- Binary wheels for all major platforms
- Battle-tested — Uvicorn depends on it for production traffic across thousands of companies
- Cython build, wraps a C library (llhttp) originally from Node.js

**Pragmatic choice.** Works today, well-maintained, proven.

### Option 2: httparse (Rust, via existing Python wrapper)

[httparse](https://github.com/seanmonstar/httparse) is the standard Rust HTTP parser — used by hyper, reqwest, actix-web, and most of the Rust web ecosystem. Zero-copy, no-alloc, push parser.

A [Python wrapper](https://github.com/adriangb/httparse) exists using PyO3, but it's a small project with unclear maintenance status. Not production-ready as-is.

### Option 3: Build our own Rust wrapper (PyO3 + maturin + httparse)

Write a thin Python extension in Rust that wraps httparse, built with PyO3/maturin.

```
plain-http-parser/         (or just inline in the server package)
├── Cargo.toml             (depends on httparse crate)
├── src/lib.rs             (PyO3 bindings — parse function + result types)
└── pyproject.toml         (maturin build config)
```

What we'd get:

- **Own the code** — fits the "own the full stack" philosophy
- **Integrate security hardening directly** — the gunicorn parser has request limits, header validation, desync protections. These could live in the Rust layer rather than being bolted on top
- **httparse has SIMD** — auto-detects SSE4.2/AVX2 for header scanning acceleration
- **Memory safe** — Rust vs C (llhttp). Both are fine in practice, but Rust's safety guarantees are real
- **Single compiled dependency we control** — no third-party release cycle to wait on
- **Small surface area** — httparse does the hard part (parsing HTTP/1.x), we just need PyO3 glue to expose "parse these bytes, give me method/path/version/headers"

What it costs:

- Rust toolchain in CI for building wheels
- Need to produce wheels for major platforms (maturin handles this well, cross-compilation via maturin-action for GitHub Actions)
- More upfront work than just depending on httptools
- We maintain it (but it's thin glue over a very stable crate)

## httparse vs llhttp

Both are extremely fast — the difference between them is negligible compared to the Python overhead of either wrapper. The meaningful comparison:

|                 | llhttp (httptools)         | httparse (Rust)                           |
| --------------- | -------------------------- | ----------------------------------------- |
| Language        | C (generated by llparse)   | Rust                                      |
| Used by         | Node.js, Uvicorn           | hyper, reqwest, actix-web                 |
| Parser model    | Callback-based push parser | Push parser returning parsed struct       |
| SIMD            | No                         | Yes (SSE4.2/AVX2 auto-detected)           |
| Memory safety   | Manual (C)                 | Guaranteed (Rust)                         |
| Python bindings | Cython (mature)            | PyO3 (mature tooling, we'd write our own) |
| Maintenance     | MagicStack                 | seanmonstar (hyper author)                |

## The lineage

```
http-parser (C, hand-written, Joyent/Node.js ~2009)
  Archived 2022, hard to maintain
       │
       ▼
   llhttp (C, generated from TypeScript via llparse, 2019)
     ~1400 lines TypeScript → compiled to C
     2.5x faster than http-parser
     Currently powers Node.js
       │
       ▼
  httptools (Cython, MagicStack ~2016)
    436 lines of Cython wrapping llhttp
    Bundles llhttp as git submodule (currently v9.3.0)
    Still uses old http-parser for URL parsing
    Used by Uvicorn

Separately:
  httparse (Rust, seanmonstar ~2015)
    Independent of Node.js lineage
    Powers hyper, reqwest, actix-web
    SIMD-accelerated (SSE4.2/AVX2)

  Milo (Rust, NearForm 2023)
    Intended to replace llhttp in Node.js
    Not yet shipped, optimizing WASM build
```

## Current architecture fit

TLS now uses asyncio transport (`loop.start_tls` with memory BIO). All connections (H1 and H2) have `asyncio.StreamReader`/`StreamWriter`. Headers are read async via `async_read_headers()` in `h1.py`, then the gunicorn parser processes the pre-buffered bytes synchronously. Small bodies parse inline on the event loop; large bodies use `AsyncBridgeUnreader` which bridges back via `asyncio.run_coroutine_threadsafe()`.

The current parser couples socket reading with parsing (pull model via `Unreader`). Both httptools and httparse are push parsers (you feed them data). This requires restructuring:

- **Current**: `async_read_headers()` → buffer bytes → `parse_request()` → gunicorn parser on buffer
- **Native**: `async_read_headers()` → feed bytes to parser → get back method/path/headers

The boundary is clean — `parse_request` in `h1.py` runs in the thread pool (for bridge mode) or inline, builds a request, and returns it. The parser's internal model doesn't leak upward. The restructuring is contained within the parse layer.

A native push parser also enables a fully feed-based model where headers are parsed incrementally as bytes arrive, rather than buffering until `\r\n\r\n`. This is a natural fit for the existing asyncio read loop.

## Recommendation

httptools is the pragmatic choice — it works today and closes the gap immediately. Building our own Rust wrapper is the principled choice — it fits the integrated server philosophy and gives us more control. Either way, the gunicorn parser's security hardening (request limits, header validation, desync protections) needs to be preserved.
